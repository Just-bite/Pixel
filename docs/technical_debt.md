# Технический долг

Под техническим долгом далее будем понимать ряд свойств, практик и особенностей протекания жизненного цикла програмного обеспечения, качественно ухудшающих его использование, поддержку и развертывание. К признакам технического долга в данной работе отнесем:
 - непонятный / нечитабельный код;
 - дублирующийся код;
 - отсутствие автоматизации (тестов, сборки, развёртывания);
 - запутанная архитектура и ненужные сложные зависимости;
 - медленные/неэффективные средства;
 - незакоммиченый код / долгоживущие ветки;
 - отсутствие / несоответствие технической документации;
 - отсутствие тестовой среды;
 - длинные циклы интеграции / отсутствие непрерывной интеграции.

Проинспектируем проект на наличие каждого из перечисленных признаков.

## 1. Непонятный / нечитабельный код
В проекте отсутствует непонятный код. Тот, что потенциально претендует на таковую позицию, либо переписывается, либо помечается комментарием. Команда следит за объемом функций и глубиной ветвления (не превышает 3).

## 2. Дублирующийся код
В коде присутсвуют некоторые повторения функциональности. В основном они связаны с неподчищенным legacy кодом, который использовался на ранних этапах проектирования.
Примером может стать класс Filemanager, функионал которого был перемещен в ProjectManager.

## 3. Отсутсвие автоматизации
Данный пункт является одним из ярковыраженных признаков технического долга проекта. Полноценные тесты, тем более автоматизированные, отсуствуют. Тестирование работы приложения на различных сценариях выполняется вручную. Сборка и развертывание выполняются на основании окружения Qt.

## 4. Запутанная архитектура и ненужные сложные зависимости
При проектировании приложения команда проекта руководствовалась принципами SOLID, DRY и KISS. Основной задачей при проектировании архитектуры являлась возможность свободного расшиерения функционала и снижение внутренних зависимостей.

## 5. Медленные / неэффективные средства
Для ускорения работы приложения был выбран язык С++, позволяющий точно манипулировать работой с выделяемой памятью. Qt среда для GUI позволяет использовать оптимизированные встроенные возможности для логирования действий, визуализации объектов, рабоы с файловой системой и многих других задач.

## 6. Незакоммиченный код / долгоживующие ветки
Команда проекта работает по принципу "1 ветка - одна фича". Каждая ветвь содержит в себе название фичи, а также дату ее создания. Все коммиты выполняются своевременно, являются содержательными. На момент спринта отсутсвуют побочные ветки кроме главной, все иные ветви объединены с разрешением конфликтов.

## 7. Отсутсвие / несоответсвие технической документации
На данном этапе техническая документация как таковая у проекта отсуствует. Это негативно сказывается на отслеживании изменений в реализуемом функционале и архитектурных решениях. Об устранении данного недостатка будет сказано позднее.

## 8. Отсутсвие тестовой среды
Отдельно выделить тестовую среду для проекта пока не представляется возможным. Тесты выполняются вручную, имитируя реальное использование програмного продукта.

## 9. Длинные циклы интеграции / отсуствие непрерывной интеграции
Слияние выполняется своевременно, при любом pull реквесте, команда постоянно активно взаимодействует и обсуждает продукт.

# Мероприятия по устранению технического долга
Исходя из перечисленных выше недостатков проекта необходимы:
1. Ведение технической документации. Для данного проекта актуальной будет не только документация для разработчика, но и руководство пользователя (запланировано как фича).
2. Внедрение Unit-тестирования (как вариант с использованием QTestLib, QBenchmark).
3. Очистка от legacy архитектурных решений.

На данном этапе объем недоимплементированных фич превалирует над объемом технического долга, однако мероприятия по устранение применить необходимо уже сейчас. Ведение документации позволит упростить дальнейшее обслуживание, тесты упростят проверку, а очистка от legacy происходит на каждой итерации разработки.
